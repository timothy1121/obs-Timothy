## 三层结构

### Input

Input——输入：不是简单收集，而是把学习到的知识化为最小知识单元，作为知识库基础，后可以重复使用。

### Internalization

Internalization——内化：不是简单理解，而是根据目的、或者任务，把相关最小知识单元组合连接起来，做成产品或工具。

### Output

Output——输出：不是简单表达，而是产出有价值的、可以重复使用的产品。

```text
输入 = Input
内化 = Internalization
输出 = Output
```

```text
目标
 ↓
输入 ←→ 内化 ←→ 输出
 ↑       ↑       ↓
 └──── 反馈与迭代 ────┘
```

## 使用方式

- 先进入 `Input`，沉淀素材、摘录、片段、最小知识单元
- 再进入 `Internalization`，围绕任务、主题、目标做组合、连接、重组
- 最后进入 `Output`，产出可复用的成果、产品、工具、方案

这三层不是一次性线性流程，而是可以来回迭代。

## 使用边界

- 这是一套母版库规则，不是具体内容库
- 母版库只保留稳定结构、说明页和模板页
- 具体知识卡、项目卡、成果卡，应放在由母版复制出去的新库里
- `.obsidian` 只保留稳定默认配置，不复制个人工作区状态

## 笔记种类

### 正式层：6 类母模板

1. `对象类`
适用：概念、术语、人物、事件

2. `机制类`
适用：原理、机制、关系、变量

3. `结构类`
适用：模型、模式、框架、时间线

4. `执行类`
适用：方法、步骤、规则、流程

5. `证据类`
适用：实验、案例、数据、指标

6. `表达类`
适用：书摘、摘录、命题、结论、假设、书籍

### 非正式层：素材

7. `素材`
用于摘录、提纲、片段、灵感、未完成内容，不升格为正式知识卡，也不参加模板考核。

## 当前使用规则

- 先判断知识类型，再决定结构
- 不再强迫所有笔记都用同一个四段式
- 一张卡只解决一个核心对象
- 保留 `关联知识`
- 中文简洁
- 机制优先
- 忠实原意
- 不写空泛口号
- 不强行凑字数

## 判断规则

- “它是什么” -> `对象类`
- “它怎么运作” -> `机制类`
- “它由什么组成” -> `结构类`
- “它怎么做” -> `执行类`
- “它证明了什么” -> `证据类`
- “它表达了什么文本或观点” -> `表达类`

## 当前治理规则

- 正式卡按模板治理
- 素材不强行套模板
- 新增笔记先分类，再套模板
- 后续只做增量治理，不做无意义的重复全库重排
